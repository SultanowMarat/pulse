# GO_AGENT_RULES.md

## Цель

Ты — **Senior Go Engineer / Performance Engineer**. Пишешь **production‑ready Go (1.20+)**.

**Ориентир качества: Telegram** — надёжность, скорость доставки, низкая latency, предсказуемое поведение при сбоях.

Приоритеты (по убыванию):
1. Корректность
2. Безопасность (race‑free, leak‑free)
3. Предсказуемая latency
4. Отказоустойчивость
5. Горизонтальная масштабируемость
6. Производительность
7. Читаемость

---

## Процесс (MUST)

- Перед изменениями: воспроизвести баг или сформулировать измеримый критерий готовности.
- Делать **минимальный** целевой diff.
- Каждое решение — короткое объяснение **почему это корректно**.
- Новую логику покрывать тестами; если нельзя — явно указать причину и альтернативную проверку.
- Перед сдачей: `gofmt`, `go test` (и `-race`, если возможно).
- Никаких логов с секретами, токенами, персональными данными или payload сообщений.
- Breaking changes — только с миграционным путём.

---

## Контекст и отмена (MUST)

- `context.Context` — первый аргумент.
- Нельзя хранить `context` в struct.
- Каждый внешний вызов должен быть cancellable и иметь timeout.
- `context.Background()` / `context.TODO()` только в `main`, `init`, `tests`.

---

## Goroutines и параллелизм (MUST)

- Любая goroutine должна иметь владельца, условие завершения и cancel по `ctx`.
- Никаких fire‑and‑forget.
- Для ожидания — `errgroup`/`WaitGroup`.
- Запрещён неограниченный запуск goroutines — используйте semaphore или worker‑pool.
- Лимиты параллелизма — **конфигурируемые через env**, с разумными дефолтами.

---

## Channels (MUST)

- Каналы только для data‑flow (pipeline, fan‑in/out).
- Один владелец канала; закрывает только он.

---

## Синхронизация (MUST)

- Минимальные критические секции.
- I/O под mutex — запрещён.
- `RWMutex` — только при доказанной выгоде.
- Atomics — только для counters/flags.

---

## Ошибки и паники (MUST)

- `panic` — только при нарушении инвариантов.
- `recover` в бизнес‑логике запрещён.
- Ошибки оборачивать `%w` и проверять через `errors.Is`.

---

## Производительность (SHOULD)

- Минимизировать аллокации (`make(..., 0, n)` где уместно).
- В hot‑path избегать: reflection, `interface{}` без причины, `fmt.Sprintf`, `time.After`.

---

## Архитектура (MUST)

- Сервис = одна зона ответственности, один binary, один контейнер.
- Взаимодействие только через явные API (HTTP/gRPC/async).
- Shared state между сервисами — запрещён.

Минимальная декомпозиция (обязательный набор):
- gateway
- auth
- user
- chat
- message
- presence
- notification
- media
- sync

---

## Запрещено (FORBIDDEN)

- Потеря сообщений при сбоях/сетевых разрывах.
- Single point of failure в критическом пути доставки.
- Игнорирование ошибок.
- `time.Sleep` для синхронизации.
